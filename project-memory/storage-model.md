# Storage Model

## Core terms
- `Conversation`: the top-level durable object stored in one folder
- `Thread`: the active linear path currently being viewed in the UI
- `Message`: one user/assistant/system/tool turn in a conversation

## Canonical layout

```text
conversation-folder/
├── conversation.yaml
├── messages/
│   ├── m0001.md
│   ├── m0002.md
│   └── ...
├── attachments/
├── exports/
└── .history/
```

## Purpose of each path

### `conversation.yaml`
Conversation-level metadata only.

Suggested fields:
- `id`
- `title`
- `created_at`
- `root_message_id`
- `active_message_id`

### `messages/*.md`
One message per file. Each file contains:
- YAML frontmatter for message metadata
- Markdown body for the visible message content

Suggested message fields:
- `id`
- `parent_id`
- `role`
- `agent`
- `timestamp`
- `format`
- `attachments`
- `revision_of` (reserved for later use if needed)

### `attachments/`
Conversation-local attachment storage. Not a hard v1 requirement beyond keeping
the directory and metadata hook in place.

### `exports/`
Readable views generated by the app for humans and external tools.

Likely early exports:
- `current.md` — active thread as Markdown
- `current.json` — optional later structured export

### `.history/`
App-managed snapshots for rollback before destructive edits/imports.

## Graph rules
- Message filenames are stable IDs, not positional ordering
- The graph is defined by message metadata, especially `id` and `parent_id`
- Inserting a message into a thread rewires parent links rather than renumbering
  files
- v1 uses single-parent messages
- Multi-parent merge nodes are a possible future extension

## Example

`conversation.yaml`

```yaml
id: conv-001
title: Agent UI planning
created_at: 2026-04-17T20:00:00-07:00
root_message_id: m0001
active_message_id: m0003
```

`messages/m0002.md`

```md
---
id: m0002
parent_id: m0001
role: assistant
agent: claude
timestamp: 2026-04-17T20:01:00-07:00
format: markdown
attachments: []
---

Use a graph with one file per message.
```

## Validation stance
- The app should validate files on load
- One malformed message file should not break the whole conversation
- The UI should surface repairable issues clearly instead of crashing
