# 2026-07-07 — Agent mode rename, CLI agent integration thinking

## Changes made

### Agent mode / auto-run rename
- Renamed "Auto-run" label to "Agent" in the ThreadView toolbar button
- Updated tooltip to explain both modes: agent mode ON = tool commands run automatically; agent mode OFF = messages forwarded to ChatGPT but no tool execution
- The underlying field name stays `auto_run` in the backend/YAML — no data migration

### Chat-interface mode fix (agent mode OFF)
The key insight: `auto_run` was gating ALL CF→ChatGPT dispatch, not just agent behaviors.
Fixed three things so chat-interface mode (agent OFF) works properly:

1. **`background.js`**: Removed the `auto_run` early-return from `checkAndDispatch`. The bridge now always dispatches pending user messages to ChatGPT regardless of agent mode. Agent mode only controls tool execution and auto-continue on the CF side.

2. **`ThreadView.tsx` refetch interval**: Polling now fires whenever `last.role === 'user'` regardless of agent mode, so ChatGPT's response appears in the UI after being synced back by the MutationObserver.

3. **`ThreadView.tsx` waiting indicator**: "Waiting for ChatGPT response" spinner now shows whenever last message is user-role, not just when agent mode is on.

What agent mode still gates (unchanged):
- Tool execution and auto-continue
- Chatbot copy appending for tool results
- The auto-run effect's main logic

## Architectural discussion saved to memory

### Storage format comparison (industry landscape)
Discussed how OpenCode uses SQLite (`~/.local/share/opencode/opencode-stable.db`) with a
`session` / `message` / `part` schema, while Codex/Claude Code use JSONL. No industry
standard exists. CF's Markdown-per-message approach is intentional: the files ARE the
export format, meant to be read by agents and humans directly.

### CLI agent integration pattern
For agents like OpenCode, Codex, and Antigravity that have their own tool execution loop,
CF acts as orchestrator — not executor. The handoff pattern:
  1. CF provides context/task (agent reads `~/Documents/context-harness/conversations/`)
  2. CLI agent does the work in its own session
  3. Agent appends a summary back to the CF conversation directory when done
  4. CF continues in the browser from there

This is a file-based handoff, not a live injection. CF stays browser-only; CLI agents
stay CLI-only. The shared state is `~/Documents/`.

## Decisions reaffirmed

- CF stays browser-focused for the foreseeable future; maximize what's possible with browser chatbots
- No change to Markdown-per-message storage format or `~/Documents/` location now
- SQLite backing layer should be reconsidered when: (a) branching is implemented and file forks become unwieldy, or (b) querying across thousands of conversations becomes a real need
