## Draft performance fix
- Split persisted editor settings into `store/settings.ts` so the hot `store/ui.ts` draft store is no longer wrapped in Zustand persist middleware.
- Root cause: `persist` runs on every store write, so every `setDraft` keystroke was triggering persistence work even though drafts were omitted from `partialize`.
- Settings remain persisted separately, while message drafts are ephemeral again.
- Verification:
  - `npm run build` -> passed with the existing CodeMirror/Vim bundle-size warning
  - `nix develop -c pytest` -> 32 passed
- Investigated persistent plain-mode editor lag. Found ThreadView still subscribed to draft store, causing the full thread with markdown/math/code rendering to re-render on every keystroke. Extracted EditorTray so draft updates only re-render the editor tray. Build and pytest passed. Do not commit until user confirms lag is fixed.

## Confirmed editor performance root cause

- User confirmed that isolating the draft subscription fixed the plain-mode lag and made Vim mode faster too.
- Final root cause was broader than CodeMirror or latex-suite: `ThreadView` subscribed to the Zustand draft value, so every keystroke re-rendered the full message thread and all expensive message rendering.
- Earlier fixes were still valid but incomplete:
    - Moving persisted editor settings out of the hot UI store removed per-keystroke persistence work.
    - Restoring `SimpleEditor` for plain mode removed CodeMirror/Vim/latex-suite from non-Vim typing.
    - Extracting `EditorTray` fixed the remaining broad-render problem by localizing the draft subscription.
- Durable decision: for hot editor state, subscribe at the smallest possible component boundary. Do not subscribe `ThreadView` or other large layout components to keystroke-level state.
- User preference captured: do not commit speculative fixes. Use WIP commits only when useful for rollback; otherwise commit after the user confirms the issue is fixed.

## Vim Escape blur debugging
- User confirmed the focus-recovery patch fixed Vim Escape behavior in Firefox.
- Debug trace showed plain `Escape` did not produce a page-visible keydown at `window`, CodeMirror `contentDOM`, or CodeMirror's handler. The only app-visible event was `focusout` from `.cm-content.cm-lineWrapping` to `body`.
- Pressing Control produced ordinary keydown logs, and a browser/content-script `inject.js` error appeared, suggesting the Escape blur was caused before Context Forge keymaps could run, likely by browser/extension behavior around contenteditable Escape handling.
- Final robust behavior: keep explicit CodeMirror Escape/Ctrl-[ handling, but also recover keyboard-induced blur by detecting editor focus moving to `body` without a recent pointer event while the document still has focus. In that case, send `<Esc>` to the Vim adapter and refocus the editor.
- Documented the debugging lesson in `project-memory/learned-lessons.md` under "Vim Escape handling can fail before application keymaps run".
