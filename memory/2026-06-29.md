## DOM export and local-model fallback discussion
- Deterministic browser adapters should be the first path for importing web
  chatbot conversations because they can target known DOM markers and produce a
  predictable Markdown transcript without model nondeterminism.
- An LLM can still be useful as a fallback when a known adapter fails or when an
  unknown chatbot UI needs best-effort extraction, but it should not be the
  default parser or canonical writer. It should propose an extracted transcript
  for review, then the normal import pipeline writes canonical conversation
  files.
- Local backend inventory from existing system config:
  Ollama runs on `127.0.0.1:11434` with `qwen3:4b` and `deepseek-r1:1.5b`
  loaded by NixOS; `llmfit` and `llama.cpp` are installed for GGUF models, with
  downloaded models stored under `~/.cache/llmfit/models/` and `llama-server`
  normally served at `http://127.0.0.1:8080/v1`.
- Prior local-model memory selected `LFM2-8B-A1B Q4_K_M` as the practical
  laptop-local GGUF model on this CPU-only Ryzen 7 PRO 5850U machine, with
  `llama-server -t 8 -c 4096` and about 6.75 tok/s generation. This is useful
  for fallback extraction experiments but likely too slow for primary agentic
  workflows.

## Context Forge as unified AI interaction harness
- User clarified the broader goal: Context Forge should become the coordination
  layer that unifies interactions across web chatbots, coding agents, local
  LLMs, and project/context files into one persistent "single brain" rather
  than just a browser chat UI.
- Existing filesystem context files already let Claude Code, Codex, and other
  agents pick up from shared project state, but that pattern has limits: web
  chatbots are excluded, LaTeX/math output is awkward, and agent outputs are not
  naturally unified into one canonical conversation thread.
- Local LLMs should be routed by task: small deterministic or low-risk fallback
  jobs can use local models; complex or high-stakes jobs can route to stronger
  agents such as Claude Code, Codex, or Antigravity. Context Forge should model
  this as adapter/provider routing rather than assuming one model backend.
